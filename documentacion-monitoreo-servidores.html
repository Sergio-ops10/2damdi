1. Identificación de Funciones Principales

A continuación se detallan los módulos de código responsables de las funciones críticas de la aplicación.

Backend y Lógica de Negocio (Python)

El archivo app.py centraliza toda la lógica.

  • Control de Servidor: Rutas @app.route que reciben peticiones HTTP y ejecutan lógica.
    @app.route('/accion/<id>/<cmd>') → Procesa encendido/apagado.
  • Conexión SSH (Paramiko): Función ejecutar_comando_ssh. Es el núcleo de la comunicación remota.
    ssh.connect(ip, user, pass) ... ssh.exec_command(cmd)
  • Gestión de VirtualBox: Uso de la librería nativa subprocess para invocar VBoxManage.

Almacenamiento de Datos

No se utiliza una base de datos SQL/NoSQL tradicional para mantener la ligereza del proyecto. En su lugar:

  • Almacenamiento en Memoria: La lista SERVIDORES en app.py actúa como base de datos volátil.
    SERVIDORES = [{ 'id': 1, 'nombre': 'Ubuntu Server 1', ... }]
  • Persistencia de Configuración: El Frontend utiliza localStorage del navegador para guardar preferencias del usuario (Tema Oscuro/Claro).

Manejo de Archivos y Recursos

  • Archivos Estáticos: Flask sirve el index.html como plantilla estática.
  • Capturas de Pantalla (Blob): La función get_vm_screenshot_bytes genera archivos temporales PNG con VBoxManage, lee sus bytes y luego elimina el archivo del disco para no generar basura.

2. Proceso de Resolución (Prompts e IAs)

El desarrollo se realizó de manera iterativa utilizando Inteligencia Artificial Generativa. A continuación se muestra la secuencia de "Prompts" (peticiones) y la evolución de la aplicación.

Etapa 1: MVP - Control Básico

IA Usada: Google Gemini 2.0 (Modelo de Razonamiento)

Prompt: "Quiero una aplicación web para controlar mis servidores Ubuntu en VirtualBox. Que pueda encenderlos y apagarlos."

Resultado: Una interfaz simple con botones. Se estableció la estructura Flask base.

Etapa 2: Monitor Real (Live View)

IA Usada: Google Gemini 2.0

Prompt: "Añade una sección para ver la pantalla de las máquinas en tiempo real."

Resultado: Implementación del endpoint de capturas de pantalla y refresco automático en JS cada 500ms.

Etapa 3: El Intento de la Terminal

IA Usada: Google Gemini 2.0

Prompt: "Quiero una terminal real en la web para escribir comandos."

Desafío: Se integró xterm.js y sesiones persistentes SSH. Sin embargo, la complejidad técnica y la UX no fueron satisfactorias.

Etapa 4: Versión Final (Métricas y Estética) -> v2.0

IA Usada: Google Gemini 2.0

Prompt: "Quita la terminal. Mejora la aplicación con gráficas de CPU/RAM, un diseño 'Premium' y modo oscuro."

Resolución Final:

  • Limpieza de código (eliminación de terminal).
  • Integración de libreria Chart.js.
  • Implementación de extracción de métricas vía SSH (parsing de top y free).
  • Refactorización CSS para Soporte de Temas.
